In the following we report a detailed description of the integration of a JDBC based catalogue.

This is a real catalogue provided by ACS which has been integrated in the SSE framework. This description reports the steps needed to expose the catalogue as Web Service compliant with the SSE framework (script development, service deployment and testing). The definition of the Schema that defines the service and the integration of the Catalogue in the SSE portal are out of the scope of this tutorial. On this subjects, please see the SSE ICD.

The Script development (1)

The script has been organized in different files according to the type of the request (search or present) and to the format of the result to be produced (hits, brief, full, browse or summary). The main script executes different scripts depending on the input parameters. In the following we report a detailed description of the main Search script, the script needed to retrieve the results in brief format. The other scripts are similar. We suppose that the Linux OS is running on the target machine and a "toolbox" user (home directory /home/toolbox/) has been created. All the files needed to deploy the service are located in the /home/toolbox/ACS directory.

Customizing the installation

The first part of the script is dedicated to the customization. All the variables that depend on the installation machine are stored in a separate script (customizeScript.xml) and are loaded (using the execute tag) when a SOAP message is received

     <setVariable name="instDir">
           <string>/home/toolbox/ACS/</string>
     </setVariable>
     <execute>
           <procedure>
                 <stringCat>
                       <variable name="instDir"/>
                       <string>customizeScript.xml</string>
                 </stringCat>
           </procedure>
     </execute>

Then other variables that define some resources (Search2SQL.xsl and TempSearch2SQL.xsl) needed during the script execution are stored in variables

      <setVariable name="xml2sql">
            <stringCat>
                  <variable name="instDir"/>
                  <string>Search2SQL.xsl</string>
            </stringCat>
      </setVariable>
      <setVariable name="txml2sql">
            <stringCat>
                  <variable name="instDir"/>
                  <string>TempSearch2SQL.xsl</string>
            </stringCat>
      </setVariable>

Extracting some variables from the request

Then the script extracts some information from the request using the xPath tag and stores them in some variables ("presentation", "cursor" and "iteratorSize")

      <setVariable name="presentation">
            <xPath xmlns:eoli="http://earth.esa.int/XML/eoli" xmlns:mass="http://www.esa.int/mass">
                  <xmlRequest/>
                  <string>mass:processSearchInputMsg/mass:searchInput/eoli:searchRequest/eoli:presentation</string>
            </xPath>
      </setVariable>
      <setVariable name="cursor">
            <xPath xmlns:eoli="http://earth.esa.int/XML/eoli" xmlns:mass="http://www.esa.int/mass">
                  <xmlRequest/>
                  <string>mass:processSearchInputMsg/mass:searchInput/eoli:searchRequest/eoli:cursor</string>
            </xPath>
      </setVariable>
      <setVariable name="iteratorSize">
            <xPath xmlns:eoli="http://earth.esa.int/XML/eoli" xmlns:mass="http://www.esa.int/mass">
                  <xmlRequest/>
                  <string>mass:processSearchInputMsg/mass:searchInput/eoli:searchRequest/eoli:iteratorSize</string>
            </xPath>
      </setVariable>

Calculating the number of results to be retrieved from the database

The database does not allow retrieving "iteratorSize" results starting from "cursor". Thus the script retrieves "cursor + iteratorSize - 1" from the database and then filters these results returning only the requested metadata. This operation is performed only if the response is different from the "hits" format (i.e. the presentation tag is not empty).

      <if>
            <oneq>
                  <variable name="presentation"/>
                  <string> </string>
      </oneq>
      <setVariable name="requestedResultSetSize">
            <itoa>
                  <plus>
                  <atoi>
                        <variable name="cursor"/>
                  </atoi>
                  <atoi>
                        <variable name="iteratorSize"/>
                  </atoi>
                  <literal value="-1"/>
                  </plus>
            </itoa>
      </setVariable>
      </if>

Connecting to the database

The script connects to the database using the con.informix.jdbc.ifxDriver driver.

The driver has to be copied on the "<TOMCAT_INSTALLATION>/webapps/TOOLBOX/WEB-INF/lib/" directory.

The connection parameters are:

  • DBURL: "jdbc:informix-sqli://www.acsys.it:1537/kes:INFORMIXSERVER=wwwInformix"
  • username: "tseug"
  • password: "guest"
      <setVariable name="connection">
            <dbConnect autoCommit="false" driver="com.informix.jdbc.IfxDriver">
                  <string>jdbc:informix-sqli://www.acsys.it:1537/kes:INFORMIXSERVER=wwwInformix</string>
                  <string>tseug</string>
                        <string>guest</string>
            </dbConnect>
      </setVariable>

The connection is stored in the "connection" variable. This variable will be used in the script to perform the operation on the database.

Defining the Region Of Interest

The Region Of Interest to include in the query string is built using a JAVA function. This function extracts the coordinates from the request (p1.lon,p1.lat p2.lon,p2.lat .... pN.lon,pN.lat), converts these coordinates in a format which is compliant with the database (p1.lon p1.lat,p2.lon p2.lat,....,pN.lon pN.lat) and builds the ROI string ST_MPolyFromText( 'multipolygon(( (p1.lon p1.lat, p2.lon p2.lat, , pN.lon pN.lat), (p1.lon p1.lat, p2.lon p2.lat, , pN.lon pN.lat), ))',4 ) to be included in the query string.

      <setVariable name="ROI">
            <methodInvocation methodName="fetch">
                  <class>
                        <classLiteral class="it.intecs.pisa.soap.toolbox.services.util.ROIFetcher2"/>
                  </class>
                  <parameter>
                        <classLiteral class="org.w3c.dom.Document"/>
                              <xmlRequest/>
                  </parameter>
            </methodInvocation>
      </setVariable>

Defining the query strings

The interaction with the database foresees two steps: firstly, a query creates a temporary table, then a second query retrieves the final results set using the temporary table. Both the query strings are built using the "xslt" tag.

The first query string is built using the ROI as input parameter

      <setVariable name="tempQueryString">
            <xslt xmlOutput="false">
                  <xmlRequest/>
                  <variable name="txml2sql"/>
                  <parameter name="roi">
                        <variable name="ROI"/>
                  </parameter>
            </xslt>
      </setVariable>

The second query string is built using the ROI, the requestedResultSetSize variables and other variables defined in the customizeScript.xml script. 

      <setVariable name="queryString">
            <xslt xmlOutput="false">
                  <xmlRequest/>
                  <variable name="xml2sql"/>
                  <parameter name="roi">
                        <variable name="ROI"/>
                  </parameter>
                  <parameter name="httpBasePath">
                        <variable name="HTTP_BASE_PATH"/>
                  </parameter>
                  <parameter                        name="databladeScaleX">
                        <variable name="DATABLADE_SCALE_X"/>
                  </parameter>
                  <parameter name="databladeScaleY">
                        <variable name="DATABLADE_SCALE_Y"/>
                  </parameter>
                  <parameter name="databladeInterp">
                        <variable name="DATABLADE_INTERP"/>
                  </parameter>
                  <parameter name="serverPath">
                        <variable name="SERVER_PATH"/>
                  </parameter>
                  <parameter name="substringStartIndex">
                        <variable name="SUBSTRING_START_IDEX"/>
                  </parameter>
                  <parameter name="requestedResultSetSize">
                        <variable name="requestedResultSetSize"/>
                  </parameter>
            </xslt>
      </setVariable>

Performing the queries

The script performs both the queries using the dbQuery tag. The first query creates the temporary table. The result is not stored

         <dbExecute>
            <variable name="connection"/>
            <variable name="tempQueryString"/>
      </dbExecute>

The second query retrieves the metadata. The result of this query is stored in the sqlResult variable

      <setVariable name="sqlResult">
            <dbQuery>
                  <variable name="connection"/>
                  <variable name="queryString"/>
            </dbQuery>
      </setVariable>

 

Executing the presentation script

The script tries to execute the script associated to the presentation format (hits, brief, full, browse, summary). If the execution is successful ("try" branch) the TOOLBOX returns the message returned by the script. Otherwise ("ifError" branch) the TOOLBOX closes the connection with the database and returns an error message compliant with the SSE ICD. The script to execute is identified concatenating the "instDir" and the "presentation" variables with the fixed string "Script.xml" (e.g. for a brief request the script path is /home/toolbox/ACS/briefScript.xml)

      <try>
            <sequence>
                  <execute>
                        <procedure>
                              <stringCat>
                                    <variable name="instDir"/>
                                    <variable name="presentation"/>
                                    <string>Script.xml</string>
                              </stringCat>
                        </procedure>
                  </execute>
            </sequence>
            <ifError errorMessageName="errorMessage">
            <sequence>
                  <dbClose>
                        <variable name="connection"/>
                  </dbClose>
                  <xml textTag="error" attributePrefix="x">
                        <mass:processSearchOutputMsg xmlns:mass="http://www.esa.int/mass">
                        <mass:searchOutput>
                        <mass:statusInfo>
                        <mass:statusId>1</mass:statusId>
                        <mass:statusMsg><error name="errorMessage"/></mass:statusMsg>
                        </mass:statusInfo>
                        </mass:searchOutput>
                        </mass:processSearchOutputMsg>
                  </xml>
            </sequence>
            </ifError>
      </try>
