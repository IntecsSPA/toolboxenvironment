Real HTTP based service

In the following we report a detailed description of the integration of a HTTP based service.

This service is one of the GIM services which have been integrated in the SSE framework. This description report the steps needed to expose the GIM Raster Clipping service as Web Service compliant with the SSE framework (script development, service deployment and testing). The definition of the Schema that defines the service, the development of the style sheet needed to integrate the service in the SSE Portal and the integration itself of the GIM Web Service in the SSE Portal is out of the scope of this tutorial. On this subject, please see the SSE ICD.

The Context

The communication between the Toolbox and the ICE services web tier is based on the HTTP request/response model. The Toolbox acts as an HTTP client for the ICE Web Server.
Incoming SOAP-messages are converted by the Toolbox into HTTP-requests and sent to the ICE Web Server. After processing the request, the ICE Web Server responds by sending XML (the response is not HTML but XML. The message returned by the Web Server can be inserted in the SOAP-response with no modification. The XML-message contains the location the result-dataset can be downloaded by the user) back to the TOOLBOX. This XML-string is used to assemble the SOAP-message that is sent back from the TOOLBOX to the SOAP Client.

The Script development

We suppose that all the resources needed to perform the service are available (c:\jwsdp-1_0_01\webapps\TOOLBOX\services\GIM_RasterClipService\extract_areaOfInterest.xsl and the ICE services is available at host="193.74.120.94", port="8123").

The service has been integrated using the Asynchronous mechanism. Thus 3 scripts are needed. The first script performs the communication with the ICE Web Server and stores the XML result file in a temporary file. The second script checks if the file has been created. The third script loads the file and send it back to the SOAP Client.

Hereafter we report a description of the main tags.

First Script

Setting the file name.

The script creates the name of a temporary file concatenating (<stringCat>) a fixed string and the <orderId>. The file will be used to store the result of the interaction with the ICE Web Server.

<sequence xmlns="http://pisa.intecs.it/mass/toolbox/xmlScript">
     <setVariable name="fileName">
          <stringCat>
               <string>c:\tmp\</string>
               <orderId/>
               <string>.xml</string>
          </stringCat>
     </setVariable>

Connecting to the ICE Web Server.

The script accesses the JSP page (/ice/raster/clip/views/xml.jsp) connecting to the ICE Web Server (193.74.120.94) on port 8123 using the <http> tag.

 <http host="193.74.120.94" port="8123">
      <url>
            <string>/ice/raster/clip/views/xml.jsp</string>
      </url>
      <parameter name="orderId">
.....
      </parameter>

The parameters needed to perform the http POST are extracted from the input message by means of the <xPath> tag

<parameter name="orderId">
      <xPath xmlns:mass="http://www.esa.int/mass">
            <xmlRequest/>
            <string>//mass:commonInput/mass:orderId</string>
      </xPath>
</parameter>
<parameter name="clippeeURL">
      <xPath xmlns:mass="http://www.esa.int/mass">
            <xmlRequest/>
            <string>//mass:sendOrderInput/mass:sourceDataset/mass:location</string>
      </xPath>
</parameter>
...
...
<parameter name="negativeValuesToZero">
      <xPath xmlns:mass="http://www.esa.int/mass">
            <xmlRequest/>
            <string>//mass:sendOrderInput/mass:negativeValuesToZero</string>
      </xPath>
</parameter>

The result of the HTTP exchange is stored in a temporary file (whose name is stored in the variable named "fileName") by means of the <dumpFile> tag.

<dumpFile>
      <variable name="fileName"/>
...
</dumpFile>

Second script

This script is executed by the TOOLBOX to check if the resources needed to create the response message are ready. It looks for the response file stored by the first script in the c:\tmp\ directory by means of the <fileExists> tag. The fileName variable has been set in the first script. The TOOLBOX supports persistence of some types of variable among the executions of XML scripts related to the same Asynchronous request. This means that a variable defined in the first Script can be referred, e.g., in the second XML Script.

<sequence xmlns="http://pisa.intecs.it/mass/toolbox/xmlScript">
      <fileExists>
            <variable name="fileName"/>
      </fileExists>
</sequence>

Third script

This script loads the response message stored by the first script in the c:\tmp\<orderId>.xml file by means of the <loadXML> tag. The fileName variable has been set in the first script.

<sequence xmlns="http://pisa.intecs.it/mass/toolbox/xmlScript">
      <loadXML>
            <variable name="fileName"/>
      </loadXML>
</sequence>
