Listing asynchronous instances

To view the list of the asynchronous instances of the selected service click on the "Display Asynchronous instances" section link in the in the "Monitoring center" page. The browser should display a table listing all the service instances

The table displayed includes the following information:

  • Instance: unique identifier assigned by the Toolbox to the instance. Clicking on the instance link the instance link it is possible to display graphical representation of an instance flow.
  • OrderId: order identifier. It is displayed only if the message is part of an order interface.
  • Operation: name of the operation.
  • Arrival Date: date in which the request has been received by the service.
  • Back-end Expiration Date: If a request is not completed within this date, the request is automatically removed from the status.
  • Push Host: IP address of the host the TOOLBOX pushes back the response to.
  • Status: status of the request. Each request can be in one of the following status:
    • waiting: the TOOLBOX has not yet executed the First script corresponding to the request. This can happen in the following two cases: the queuing mechanism is on and the request has not been scheduled yet; the service is suspended in SOFT mode;
    • rejected: the request arrived to the TOOLBOX but the service is suspended in HARD mode.
    • executing: the TOOLBOX is executing the First script corresponding to the request.
    • pending: the TOOLBOX has successfully executed the First script corresponding to the request.
    • aborted: the execution of the First script from the TOOLBOX returned an error.
    • cancelled: the request has been manually cancelled by the TOOLBOX administrator using the TOOLBOX ADMINISTRATOR Web Application
    • expired: the request timeout has elapsed and the TOOLBOX has not yet executed the Second script corresponding to the request or the Second script has not yet returned "true".
    • responseLeaving: this status can be reached by a request only if the "retry" mechanism is on. A request is in status "responseLeaving" if the corresponding response is ready to be pushed to the Portal, but pushing continues failing. The TOOLBOX tries to push the response back to the Portal a number of times given by the Operation "retry attempts" parameter. If the TOOLBOX fails to send back the response, the request is set to unpushed.
    • error: the execution of the Third script from the TOOLBOX returned an error.
    • unpushed: the TOOLBOX was not able to send back the response (successful or error) to the Portal. The un-pushed response is stored on disk and is accessible via the instance flow page.
    • completed: the TOOLBOX was able to send back the response (successful or error) to the Portal. The pushed response is stored on disk and is accessible via the instance flow page.

  • setting the request in the status "cancelled" by clicking on the "cancel" button. When a request is set in status "cancelled" the TOOLBOX stops its processing and pushes back to the portal an error response claiming that the request has been cancelled.
  • Deleting an instance which is in "completed" or "error" status. All the resources related to this instance (log files, input/output messages, error emails) will be deleted from the file system.
