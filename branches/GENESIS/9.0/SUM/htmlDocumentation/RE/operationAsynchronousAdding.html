Adding an asynchronous operation

To add a new asynchronous operation follow these steps.

Display the list of the Service operations available.

Click on the "ADD Operation" button the browser should display

In the "Choose the operation name" enter the mane of the operation. The Operation Name is only a mnemonic name which is used by the TOOLBOX to store and display information related to a specific operation. It is not used to deploy the real operation and to create the WSDL file. To publish a SSE or EOLI compliant operation select the operation name using the drop down list. For "generic operation" (operation which are not compliant with the SSE ICD or with the EOLI ICD) deployment select the "Other operation:" radio box and enter the operation name.

In the "Choose the communication mode the new operation will support:" select the Asynchronous mode option of radio box.

Click on "Next >>" the browser will display asynchronous operation page. By clicking on the "<< Back" button you will be redirected to the Operation listing page.

fill the form as follow:

Operation specification

  • SOAP Action (only for operations that are not compliant with SSE or EOLI): enter the SOAP Action that will be associated to the operation. The SOAP Action is used by the TOOLBOX to link the scripts to a specific SOAP message sent to a Service installed in the TOOLBOX. This parameter will be used to create the WSDL file. For SSE or EOLI operation this input parameter is disabled. It is automatically set by the TOOLBOX.
  • First Operation Script file: click on the "Browse" button and select the script file that will be executed when a SOAP message associated to the SOAP Action will be received by the Service installed in the TOOLBOX.
  • Second Operation Script file: click on the "Browse" button and select the script file that will be executed by the TOOLBOX to check if all the resources to build the response (e.g. the resources needed to execute the third script) are available. Remember that this script has to return a "boolean".
  • Third Operation Script file: click on the "Browse" button and select the script file that will be executed by the TOOLBOX push manager when the second script has been successfully executed. Remember that this script has to return an XML SOAP payload that will be sent back to the end point included in the WS-Addressing header of the SOAP message associated to the "Operation".
Asynchronous communication settings
  • Push Admitted Hosts: enter a list of the hosts allowed to send/receive requests to/from the service. If the field has left blank the SOAP requests coming from any host will be served). When the list is not empty, the TOOLBOX pushes back the SOAP response message to the hosts (the complete URL is extracted from the WS-Addressing SOAP header) that are included in the list.
  • Request timeout: enter the timeout of the request coming from a client. It is the time elapsed since the request arrival and the removal of the request from the status of the service.
  • Polling Rate: enter the rate the TOOLBOX push manager executes the second Operation Script.
  • Retry on failing pushing response: The service can be configured (by ticking the checkbox) such that, in case of failing pushing a response, the TOOLBOX tries to push again and again for a configurable (by means of the "Attempts" field) amount of times and a configurable (by means of the "Rate" field) rate.

Click on the "Create". If all the parameters are valid you will be redirected to the Operation list page. The list should now include the new operation just created. If some parameters are not valid or missing, you will be redirected to the same page and a WARN message will outline the missing or wrong parameter.
